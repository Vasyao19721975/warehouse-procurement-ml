# ML System Design Doc

## 1. Problem Definition

### 1.1 Business Problem

Компания сталкивается с проблемой неэффективного управления складскими запасами.

Основные проблемы:
- частые ситуации out-of-stock (дефицит товаров)
- избыточные запасы (переполнение склада)
- ручное планирование закупок

Это приводит к:
- потере выручки
- увеличению затрат на хранение
- неэффективному использованию капитала

---

### 1.2 Objective

Цель — разработать ML-систему, которая автоматизирует процесс принятия решений по закупкам.

Система должна:
- прогнозировать спрос на товары
- учитывать текущие остатки
- формировать рекомендации по закупкам

---

### 1.3 Success Metrics

Метрики эффективности:

- MAE (ошибка прогноза)
- снижение out-of-stock
- снижение избыточных запасов

---

### 1.4 Assumptions

- исторические данные отражают будущий спрос
- данные по продажам корректны
- нет сильных внешних факторов (акции, сезонность)

---

### 1.5 Constraints

- ограниченное количество данных
- отсутствие внешних факторов
- возможный шум в данных


## 2. Data

### 2.1 Data Sources

В системе используются следующие источники данных:

1. Данные продаж:
- sku_id
- date
- sales (количество проданных единиц)

2. Данные остатков:
- sku_id
- date
- stock

3. Данные поставок:
- sku_id
- date
- supply

---

### 2.2 Data Description

Данные представляют собой временные ряды по каждому SKU.

Каждая запись содержит информацию о состоянии товара в определённый день:
- уровень продаж
- текущий остаток
- поступления

---

### 2.3 Data Preprocessing

Этапы обработки данных:

- приведение типов данных (date → datetime)
- сортировка по sku_id и дате
- удаление пропусков
- агрегация данных по дням

---

## 3. Feature Engineering

Для обучения модели используются признаки:

- lag_1 — продажи за предыдущий день
- lag_2 — продажи за 2 дня назад

Дополнительно:
- текущий уровень stock
- наличие поставок (supply)

---

## 4. Modeling

### 4.1 Model Choice

Используется модель:

- RandomForestRegressor

Причины выбора:
- устойчива к шуму
- не требует масштабирования
- хорошо работает на табличных данных

---

### 4.2 Training

Модель обучается на исторических данных продаж.

Целевая переменная:
- прогноз продаж (sales)

---

## 5. Evaluation

### 5.1 Metric

Используется метрика:

- MAE (Mean Absolute Error)

Причины:
- интерпретируемость
- показывает среднюю абсолютную ошибку прогноза

---

### 5.2 Validation

Разделение данных:
- обучение на исторических данных
- тестирование на более поздних датах

---

## 6. Output

Результат используется для поддержки решений по закупкам:

Файл:
- final_recommendations.csv

Содержит:
- sku_id
- прогноз спроса
- рекомендацию:
  - increase_order
  - normal_order
  - no_order

## 7. System / Architecture

### 7.1 General Approach

Система реализована как batch-пайплайн для регулярного пересчёта рекомендаций по закупке товаров.

Основной сценарий работы:
1. загрузка данных об остатках и поставках;
2. предобработка данных;
3. расчёт продаж;
4. построение baseline-рекомендаций;
5. обучение ML-модели;
6. прогнозирование спроса;
7. формирование итоговых рекомендаций;
8. сохранение результатов.

---

### 7.2 Pipeline Stages

Система состоит из следующих этапов:

#### Stage 1. Data Ingestion
На этом этапе система считывает сырые данные из файлов:
- остатки товаров;
- поставки товаров.

#### Stage 2. Data Preprocessing
Данные очищаются, приводятся к нужным типам и агрегируются по SKU и дате.

#### Stage 3. Sales Calculation
На основе изменения остатков и поставок рассчитываются продажи по каждому товару.

#### Stage 4. Baseline Recommendation
На основе средней скорости продаж вычисляются:
- sales_per_day
- days_of_stock
- recommended_order
- decision

#### Stage 5. ML Training
На подготовленных исторических данных формируются признаки:
- lag_1
- lag_2

После этого обучается модель RandomForestRegressor.

#### Stage 6. Prediction
Модель прогнозирует спрос для каждого SKU.

#### Stage 7. Final Recommendation
На основе прогноза спроса рассчитывается:
- predicted_sales
- ml_recommended_order
- difference между baseline и ML-подходом

#### Stage 8. Output Saving
Результат сохраняется в итоговый файл:
- outputs/final_recommendations.csv

---

### 7.3 Main Components

Основные компоненты системы:

- `load_data.py` — загрузка данных;
- `preprocess.py` — очистка и подготовка данных;
- `calculate_sales.py` — расчёт продаж;
- `recommendation_pipeline.py` — базовые рекомендации;
- `ml_pipeline.py` — обучение и прогнозирование;
- `main.py` — оркестрация полного пайплайна;
- `config.py` — централизованное хранение настроек и путей.

---

### 7.4 Execution Model

Система запускается в batch-режиме.

Команда запуска:
```bash
python -m src.main
```


## 8. Service Layer (Batch Service)

### 8.1 Service Overview

Система может быть оформлена как batch-сервис для автоматического расчёта рекомендаций по закупке.

Сервис выполняет полный pipeline:
- загрузка данных
- предобработка
- расчёт продаж
- построение baseline-рекомендаций
- обучение модели
- прогнозирование
- формирование итоговых рекомендаций

---

### 8.2 Run Strategy

Сервис может запускаться:

- вручную (через CLI)
- по расписанию (ежедневно / еженедельно)
- при обновлении данных

Пример запуска:

```bash
python -m src.main
```

---

### 8.3 Production Setup (Future)

В production-сценарии система может быть:

- обёрнута в Airflow DAG
- запускаться по cron
- хранить результаты в базе данных
- интегрироваться с BI-системами

---

### 8.4 Output Usage

Результаты используются:

- закупщиками
- аналитиками
- системами планирования

Файл:
Файл:
- outputs/final_recommendations.csv

---

### 8.5 Why Batch?

Batch-подход выбран потому что:

- закупки принимаются не в real-time
- данные обновляются периодически
- нет необходимости в онлайн-инференсе

---

### 8.6 Future Improvements

- переход к near real-time
- API сервис
- мониторинг качества модели
- автоматическое переобучение
