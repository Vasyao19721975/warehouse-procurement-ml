# ML System Design Doc

## 1. Problem Definition

### 1.1 Business Problem

Компания сталкивается с проблемой неэффективного управления складскими запасами.

Основные проблемы:
- частые ситуации out-of-stock (дефицит товаров)
- избыточные запасы (переполнение склада)
- ручное планирование закупок

Это приводит к:
- потере выручки
- увеличению затрат на хранение
- неэффективному использованию капитала

---

### 1.2 Objective

Цель — разработать ML-систему, которая автоматизирует процесс принятия решений по закупкам.

Система должна:
- прогнозировать спрос на товары
- учитывать текущие остатки
- формировать рекомендации по закупкам

---

### 1.3 Success Metrics

Метрики эффективности:

- MAE (ошибка прогноза)
- снижение out-of-stock
- снижение избыточных запасов

---

### 1.4 Assumptions

- исторические данные отражают будущий спрос
- данные по продажам корректны
- нет сильных внешних факторов (акции, сезонность)

---

### 1.5 Constraints

- ограниченное количество данных
- отсутствие внешних факторов
- возможный шум в данных


## 2. Data

### 2.1 Data Sources

В системе используются следующие источники данных:

1. Данные продаж:
- sku_id
- date
- sales (количество проданных единиц)

2. Данные остатков:
- sku_id
- date
- stock

3. Данные поставок:
- sku_id
- date
- supply

---

### 2.2 Data Description

Данные представляют собой временные ряды по каждому SKU.

Каждая запись содержит информацию о состоянии товара в определённый день:
- уровень продаж
- текущий остаток
- поступления

---

### 2.3 Data Preprocessing

Этапы обработки данных:

- приведение типов данных (date → datetime)
- сортировка по sku_id и дате
- удаление пропусков
- агрегация данных по дням

---

## 3. Feature Engineering

Для обучения модели используются признаки:

- lag_1 — продажи за предыдущий день
- lag_2 — продажи за 2 дня назад

Дополнительно:
- текущий уровень stock
- наличие поставок (supply)

---

## 4. Modeling

### 4.1 Model Choice

Используется модель:

- RandomForestRegressor

Причины выбора:
- устойчива к шуму
- не требует масштабирования
- хорошо работает на табличных данных

---

### 4.2 Training

Модель обучается на исторических данных продаж.

Целевая переменная:
- прогноз продаж (sales)

---

## 5. Evaluation

### 5.1 Metric

Используется метрика:

- MAE (Mean Absolute Error)

Причины:
- интерпретируемость
- показывает среднюю абсолютную ошибку прогноза

---

### 5.2 Validation

Разделение данных:
- обучение на исторических данных
- тестирование на более поздних датах

---

## 6. Output

Результат работы системы:

Файл:
- final_recommendations.csv

Содержит:
- sku_id
- прогноз спроса
- рекомендацию:
  - increase_order
  - normal_order
  - no_order
